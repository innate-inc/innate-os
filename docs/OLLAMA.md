# Semantic Skill Analyzer — ollama integration

The `semantic_skill_analyzer` package uses a local [ollama](https://ollama.com) instance to cross-reference a natural-language scenario against the robot's existing capabilities and produce a structured JSON file describing what is already covered and what needs to be built.

## Prerequisites

| Requirement | Notes |
|-------------|-------|
| ollama running locally | `ollama serve` (default port 11434) |
| `qwen3:0.6b` model pulled | `ollama pull qwen3:0.6b` |
| `ollama` Python package | included in `ros2_ws/pip-requirements.txt`; already present in the `local_llm` conda env if developing locally |
| `~/.wildrobot/capabilities.json` | generated by `scripts/export_capabilities.py` or automatically on robot restart |

## How it works

1. `analyze()` loads `~/.wildrobot/capabilities.json` — the index of all agents and their skills.
2. It sends two inputs to the LLM: the user scenario and the capabilities JSON.
3. The LLM is given a tool (`save_capability_analysis`) that enforces a typed output schema, avoiding free-form text parsing.
4. The result is written to `~/.wildrobot/<uuid>-missing-skills.json` and the path is returned.

## Output format

```json
{
  "existing_agents": ["demo_agent", "security_guard_agent"],
  "missing_capabilities": {
    "visitor_monitor_agent": {
      "prompt": "Detailed description of what this agent does and which skills it calls.",
      "new_skills": [
        {"detect_face": "Detect and identify a human face using the robot camera."},
        {"log_arrival": "Write a timestamped visitor entry to a local log file."}
      ]
    }
  }
}
```

- **`existing_agents`** — agents already in the system that can handle this request fully or partially.
- **`missing_capabilities`** — new agents that need to be created, each with a detailed prompt and a list of new skills they require. Empty when everything is already covered.

## Python API

```python
from semantic_skill_analyzer import analyze, load_capabilities

# Run gap analysis — returns path to the written file
output_path = analyze("I want the robot to greet visitors and log their arrival time")

# Inspect the result
import json, pathlib
result = json.loads(pathlib.Path(output_path).read_text())
print(result["existing_agents"])       # agents that already cover the request
print(result["missing_capabilities"])  # new agents/skills to build

# Load the capabilities index directly (no LLM call)
caps = load_capabilities()  # reads ~/.wildrobot/capabilities.json
```

### `analyze()` parameters

| Parameter | Type | Default | Description |
|-----------|------|---------|-------------|
| `prompt` | `str` | required | Natural-language scenario to analyze |
| `capabilities_path` | `str` or `Path` | `~/.wildrobot/capabilities.json` | Capabilities index to compare against |
| `output_dir` | `str` or `Path` | `~/.wildrobot` | Directory for the output file |
| `ollama_host` | `str` | `http://localhost:11434` | Ollama API endpoint |

Returns the absolute path of the written `<uuid>-missing-skills.json` file.

## Test scripts

### Script 1 — gap analysis (produces a missing-skills file)

Use this when you want to explore what new agents or skills need to be built for a scenario the robot cannot yet handle.

```bash
conda run -n local_llm python3 scripts/test_skill_gap_analysis.py
# or with a custom prompt:
conda run -n local_llm python3 scripts/test_skill_gap_analysis.py "Monitor the corridor and alert via Telegram if an unknown person is detected"
```

Expected output:
```
Covered by existing agents:
  ✓ security_guard_agent

Missing capabilities:

  [New agent] corridor_monitor_agent
  Prompt: Continuously monitors the corridor camera feed ...
  New skills:
    • identify_known_person: Compare detected face against a stored registry ...
    • send_telegram_photo: Capture the current camera frame and send it ...
```

### Script 2 — coverage check (verifies existing capabilities)

Use this to confirm which existing agents can handle a request that is already supported.

```bash
conda run -n local_llm python3 scripts/test_capabilities_covered.py
# or with a custom prompt:
conda run -n local_llm python3 scripts/test_capabilities_covered.py "Navigate to position 1.5, 2.0 and wave"
```

Expected output:
```
✓ Request is fully covered by existing agents:
    demo_agent
```

## Integration with innate-os components

Since `INNATE_OS_ROOT` is added to `sys.path` at runtime by `agent_loader.py`, any innate-os component can import the library directly:

```python
from semantic_skill_analyzer import analyze

output_path = analyze(user_prompt)
```

No additional path setup is required when running inside the ROS environment.

## Updating the capabilities index

The capabilities index is regenerated automatically every time the robot restarts (via `initialize_agents()` in `brain_client/initializers.py`). To regenerate it manually without restarting:

```bash
# With ROS workspace sourced (full skill descriptions):
python3 scripts/export_capabilities.py

# Without ROS (skill basenames only, no descriptions):
conda run -n local_llm python3 scripts/export_capabilities.py
```

Previous snapshots are preserved as `capabilities.json.<ISO8601>` in `~/.wildrobot/`.

## Notes on model quality

`qwen3:0.6b` is a 600M-parameter model — fast and lightweight but limited in reasoning depth. For more accurate gap analysis, pull and use a larger variant:

```bash
ollama pull qwen3:4b
```

Then change `MODEL` in `semantic_skill_analyzer/analyzer.py`:

```python
MODEL = "qwen3:4b"
```

No other code changes are needed.
